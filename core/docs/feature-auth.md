# Feature: Authentication

Cuan's authentication is powered by **Better Auth** (with the Drizzle adapter). It provides robust session management and credential handling directly within the Elysia backend.

## Context & Rationale
We chose **Better Auth** because it is framework-agnostic, has first-class Drizzle ORM integration, and cleanly separates auth logic without forcing Next.js or heavy Express middleware on us. 
Sessions are cookie-based rather than JWTs, allowing for easier session invalidation and stronger server-side control over active devices.

## Architecture

- **Library:** Better Auth (`better-auth` and `@better-auth/cli`).
- **Storage:** PostgreSQL (via Drizzle ORM). The required tables (`user`, `session`, `account`, `verification`) are generated by the Better Auth CLI and managed through standard Drizzle migrations.
- **Provider:** Email and Password (configured via `emailAndPassword: { enabled: true }`).

## Key Components

### 1. Configuration (`modules/auth/index.ts`)
The `betterAuth` instance is configured here. It hooks into the Drizzle database singleton and uses environment variables (`BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`) for security and routing.

### 2. Elysia Mount
Better Auth exposes a standard web handler. Because Elysia runs natively on Bun, this handler is mounted directly in `app.ts` (or `index.ts`):
```ts
app.mount('/api/auth', auth.handler)
```
This automatically handles routes like `/api/auth/sign-in`, `/api/auth/sign-up`, `/api/auth/sign-out`, and `/api/auth/get-session`.

### 3. Auth Guard (`modules/auth/auth-guard.ts`)
To protect private routes, an Elysia macro/plugin (`authGuard`) is used.
- It intercepts requests and reads the session cookie/headers.
- It calls `auth.api.getSession({ headers: request.headers })`.
- If a session is valid, it injects the `user` and `session` objects into the request context.
- If invalid or absent, it halts the request and returns a `401 Unauthorized` response.

## Database Schema (`auth.schema.ts`)
The auth tables are maintained in the `auth` module schema and re-exported centrally:
- `user`: Stores basic user details (id, name, email, emailVerified, image, timestamps).
- `session`: Stores active session tokens (id, userId, token, expiresAt, ipAddress, userAgent).
- `account`: Stores OAuth linkages (if added later).
- `verification`: Stores email verification tokens.

## Known Gotchas

- **Header Extraction:** Elysia is built on Web Standard Requests. When checking sessions inside the `authGuard`, we MUST manually extract headers from the `request` context and pass them to `auth.api.getSession({ headers: request.headers })`. If omitted, Better Auth won't find the cookie.
- **Better Auth CLI:** Do NOT manually modify the `auth` tables in `schema.ts`. If you add OAuth providers or fields later, use `@better-auth/cli generate` and copy the output into our schema file.
